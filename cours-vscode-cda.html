01

Qu'est-ce que VSCode vraiment ?

VSCode est souvent présenté comme un simple éditeur de texte amélioré. C'est beaucoup plus que ça. Il repose sur Electron (Chrome + Node.js), ce qui signifie que l'application tourne en réalité dans un navigateur web embarqué. Ça explique pourquoi les extensions sont écrites en TypeScript/JavaScript, et pourquoi certaines choses (comme les performances sur de très gros projets) ont leurs limites.

📌 VSCode vs Visual Studio
Ne pas confondre les deux. Visual Studio est un IDE lourd de Microsoft (plusieurs Go), principalement pour .NET/C#. VSCode est léger, gratuit, multi-langage, open-source. Tu utiliseras VSCode pour Java, HTML/CSS/JS, et même du C# avec les bonnes extensions. Pour certains projets .NET complexes (WPF, WinForms, Xamarin), Visual Studio reste nécessaire.

Architecture : comment ça marche sous le capot

VSCode sépare en deux processus distincts :

🖼

Renderer Process

L'interface visuelle. Ce que tu vois. Tourne dans Electron/Chromium.

⚙️

Extension Host

Process isolé qui exécute les extensions. Un crash d'extension ne plante pas l'éditeur.

🔌

Language Server Protocol

Les extensions de langage (Java, C#…) communiquent via LSP — un protocole standardisé.

🔍

Debug Adapter Protocol

Le débogueur parle au langage cible via DAP — même principe que LSP.

💡 Pourquoi c'est important
Comprendre l'architecture LSP explique pourquoi il faut parfois attendre quelques secondes au démarrage : l'extension lance un serveur de langage en arrière-plan (ex : le Java Language Server). Si ton IntelliSense ne fonctionne pas, c'est souvent que le language server n'a pas encore fini de s'initialiser.

📁

Explorer

Arborescence de fichiers, gestion du projet. Glisser-déposer, renommer, supprimer.

🔎

Search

Recherche globale dans tous les fichiers. Supporte les regex. Indispensable.

🌿

Commits, branches, diffs visuels — directement dans l'IDE.

▶️

La Status Bar (barre du bas)

La barre de statut est bien plus qu'un affichage. Chaque élément est cliquable :

Éléments cliquables de la Status Bar
Branche Git (ex: main) → changer de branche ou créer une nouvelle · Erreurs/Warnings (🔴 0 ⚠️ 2) → ouvre les Problems · Langage (ex: Java) → forcer la coloration syntaxique · Encodage (ex: UTF-8) → reencoder le fichier · Fin de ligne (CRLF/LF) → crucial en équipe, Git peut signaler des diffs à cause de ça
⚠️ Windows vs Linux/Mac : les fins de ligne
Windows utilise CRLF (\r\n), Linux/Mac utilisent LF (\n). En équipe sur Git, c'est une source de conflits silencieux. Configurer files.eol dans settings.json ou utiliser un .editorconfig à la racine du projet.
03

L'éditeur et IntelliSense

IntelliSense : plus qu'une autocomplétion

IntelliSense est le nom générique de Microsoft pour un ensemble de fonctionnalités d'assistance à la saisie. Il ne se résume pas à proposer des mots : il analyse le contexte, les types, les importations disponibles.

🧠

Complétion contextuelle

Propose des méthodes/propriétés selon le type réel de l'objet.

📖

Parameter hints

Affiche la signature de la méthode en cours de saisie des arguments.

🔍

Quick Info

Hover sur un symbole → documentation inline, type, fichier source.

📦

Auto-imports

Ajoute automatiquement l'import manquant en acceptant la suggestion.

💡 Déclencher manuellement IntelliSense
Si la suggestion n'apparaît pas : Ctrl+Espace. Pour les hints de paramètres : Ctrl+Shift+Espace. Pour naviguer dans les suggestions : ↑ ↓, valider avec Tab ou Entrée.

Navigation dans le code

VSCode transforme la navigation en quelque chose de fluide. Ces actions changent la façon de travailler :

🚀 Les actions de navigation essentielles
Go to Definition F12 — saute vers la déclaration du symbole sous le curseur, même dans un autre fichier ou une bibliothèque.

Peek Definition Alt+F12 — affiche la définition en inline sans quitter le fichier courant.

Find All References Shift+F12 — liste tous les endroits où ce symbole est utilisé dans le projet.

Rename Symbol F2 — renomme intelligemment un symbole partout dans le projet (pas un simple Ctrl+H).

Refactoring et actions de code

Quand une ampoule 💡 apparaît à gauche d'une ligne, c'est que VSCode (ou une extension) propose une Quick Action. Déclencher avec Ctrl+. ou cliquer sur l'ampoule. Exemples typiques :

Exemples de Quick Actions selon le langage
En Java : extraire une méthode, implémenter une interface, générer getters/setters, ajouter un import.
En C# : générer un constructeur, extraire une interface, convertir en expression lambda.
En HTML/CSS : extraire du CSS inline vers une règle, wrapper un élément dans un tag.

Minimap et breadcrumbs

La minimap (côté droit de l'éditeur) donne une vue aérienne du fichier. Elle affiche aussi les erreurs et les résultats de recherche en surbrillance. Elle peut être désactivée si tu travailles sur un petit écran (editor.minimap.enabled: false).

Les breadcrumbs (fil d'Ariane en haut de l'éditeur) affichent le chemin fichier → classe → méthode → élément actuel. Cliquables pour naviguer rapidement dans la structure.

04

Le terminal intégré

Le terminal intégré n'est pas qu'une commodité — c'est l'environnement depuis lequel tu lanceras des compilateurs, des serveurs, Maven/Gradle pour Java, MSBuild pour .NET, npm pour le web, et tous tes scripts Git.

Profils de terminal

VSCode supporte plusieurs profils de terminal en parallèle. Sur Windows, tu peux avoir PowerShell, Git Bash et WSL (Ubuntu) en simultané — chacun dans un onglet. Pour changer le profil par défaut : Ctrl+Shift+P → Terminal: Select Default Profile.

💡 Plusieurs terminaux en parallèle
Clique sur le + dans le terminal pour ouvrir un second terminal. Tu peux en avoir autant que tu veux — un pour ton serveur Java, un pour npm watch, un pour Git. Chaque terminal garde son historique et son répertoire courant indépendamment.

Le terminal comprend le projet

Quand tu ouvres un terminal depuis VSCode, il se positionne automatiquement à la racine du workspace. Plus besoin de naviguer jusqu'au projet avec cd. Les variables d'environnement définies dans le workspace sont aussi injectées automatiquement.

⚠️ Encoding et PowerShell
Sur Windows, PowerShell peut afficher des caractères bizarres avec certaines commandes UTF-8. Si tu rencontres ça avec Java ou Maven : ajouter $OutputEncoding = [Console]::OutputEncoding = [Text.Encoding]::UTF8 dans ton profil PowerShell, ou passer sur Git Bash.
05

Git & contrôle de sources

VSCode dispose d'une intégration Git native très complète. Tu n'auras pas toujours besoin de la ligne de commande — mais il faut comprendre ce que l'interface fait réellement.

Le Gutter Git (la gouttière colorée)

Dans la gouttière (à gauche des numéros de ligne), des barres colorées apparaissent automatiquement quand le fichier fait partie d'un dépôt Git :

🟢

Ligne ajoutée

Cette ligne n'existait pas dans le dernier commit. Nouveau code.

🔵

Ligne modifiée

Ligne existante dont le contenu a changé depuis le dernier commit.

🔴

Ligne supprimée

Indique qu'une ou plusieurs lignes ont été supprimées à cet endroit.

Cliquer sur une barre colorée affiche un diff inline et propose des actions : Stage Hunk, Revert Hunk, View Diff. C'est extrêmement utile pour ne stager que certaines modifications d'un fichier.

Workflow Git dans VSCode

Le panneau Source Control (Ctrl+Shift+G) présente :

Structure du panneau Source Control
Changes → fichiers modifiés non stagés (équivalent git status)
Staged Changes → fichiers prêts à être commités (après git add)
Merge Changes → fichiers en conflit lors d'un merge

Le champ texte en haut = message de commit → Ctrl+Entrée pour commiter.

GitLens (extension recommandée)

L'extension GitLens enrichit massivement l'intégration native : elle affiche sur chaque ligne qui a modifié ce code et quand (git blame inline), permet de visualiser l'historique d'un fichier ou d'une fonction, et de comparer des branches visuellement. Incontournable dès que tu travailles en équipe.

🔑 Pratique recommandée dès le début
Prendre l'habitude de faire des commits fréquents et granulaires depuis le début de la formation. Un commit par fonctionnalité, pas un gros commit en fin de journée. VSCode facilite ça avec le staging par hunk (morceau de code) plutôt que par fichier entier.

07

Configuration et settings.json

VSCode a deux niveaux de configuration : les User Settings (global, pour tous tes projets) et les Workspace Settings (spécifique à un projet, stocké dans .vscode/settings.json à la racine). Les workspace settings surchargent les user settings.

Accéder aux settings

3 façons d'accéder aux paramètres
Interface graphique : Ctrl+, — pratique pour explorer
JSON direct : Ctrl+Shift+P → Open User Settings (JSON) — plus puissant
Workspace : créer/modifier .vscode/settings.json à la racine du projet

Configuration recommandée

Voici une base solide de settings.json pour ta formation :

{
  // Formatage automatique à la sauvegarde
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",

  // Indentation (Java/C# : 4 espaces ; JS/HTML : 2)
  "editor.tabSize": 4,
  "editor.insertSpaces": true,

  // Affichage
  "editor.wordWrap": "on",
  "editor.renderWhitespace": "boundary",
  "editor.bracketPairColorization.enabled": true,
  "editor.guides.bracketPairs": true,

  // Fichiers
  "files.eol": "\n",
  "files.trimTrailingWhitespace": true,
  "files.insertFinalNewline": true,

  // Terminal
  "terminal.integrated.defaultProfile.windows": "Git Bash",

  // Git
  "git.confirmSync": false,
  "git.autofetch": true
}

Le dossier .vscode

Au-delà de settings.json, le dossier .vscode/ peut contenir :

Fichiers dans .vscode/
settings.json — paramètres du workspace (à commiter dans Git pour partager la config en équipe)
launch.json — configurations de lancement/debug (Java, .NET, Node...)
tasks.json — tâches automatisées (build, run, lint, tests)
extensions.json — extensions recommandées pour le projet (s'affichent automatiquement à l'ouverture)
08

Le débogage

Le débogueur de VSCode est l'un des plus puissants disponibles dans un éditeur. Il repose sur le Debug Adapter Protocol (DAP) — chaque extension de langage fournit son propre adaptateur.

Les breakpoints

Un breakpoint est une instruction "pause ici pendant l'exécution". Cliquer dans la gouttière à gauche d'un numéro de ligne pour en placer un. La ligne devient marquée en rouge. À l'exécution en mode debug, le programme s'arrête sur cette ligne et tu peux inspecter l'état de la mémoire.

🔴

Breakpoint simple

Pause inconditionnelle. Clic dans la gouttière.

🟡

Conditional breakpoint

Pause seulement si une condition est vraie. Clic droit sur le breakpoint.

🔢

Hit count

Pause après N passages sur cette ligne. Utile dans les boucles.

📝

Logpoint

Affiche un message dans la console sans stopper l'exécution. Alternative au console.log.

Les actions de débogage

🎮 Contrôles de débogage
F5 Continue — reprend l'exécution jusqu'au prochain breakpoint
F10 Step Over — exécute la ligne courante, passe à la suivante (sans entrer dans les fonctions)
F11 Step Into — entre dans la fonction appelée sur la ligne courante
Shift+F11 Step Out — sort de la fonction courante, remonte dans l'appelant
Ctrl+Shift+F5 Restart — relance la session de debug
Shift+F5 Stop — arrête la session de debug

La vue Variables et Watch

Quand le programme est en pause, le panneau de gauche affiche :

Panneaux disponibles en mode debug
Variables — état de toutes les variables dans le scope courant, leur type et leur valeur. Les objets sont explorables.
Watch — expressions personnalisées à surveiller (ex: myList.size(), user.isAuthenticated())
Call Stack — pile d'appels complète, cliquable pour naviguer dans les frames
Breakpoints — liste de tous les breakpoints actifs, désactivables individuellement

launch.json : configurer le debug

Pour des configurations répétables (arguments, variables d'env, ports), créer un .vscode/launch.json. VSCode propose un assistant à la création via le bouton "Créer un fichier launch.json" dans le panneau Run & Debug.

09

Workspace et multi-projet

Un workspace dans VSCode, c'est soit un dossier ouvert, soit un fichier .code-workspace qui regroupe plusieurs dossiers. Pendant ta formation, tu ouvriras souvent des dossiers de projet directement — c'est le cas le plus simple.

Multi-root workspace

Pour les projets avec un back-end Java et un front-end JS séparés par exemple, on peut ouvrir plusieurs dossiers dans le même workspace via File → Add Folder to Workspace. Chaque dossier conserve ses propres settings, son propre Git, mais tout est visible dans un seul Explorer.

💡 Ouvrir VSCode depuis le terminal
La commande code . ouvre VSCode dans le répertoire courant. code fichier.java ouvre directement un fichier. C'est la façon la plus rapide de travailler — naviguer vers ton projet dans le terminal, puis code ..

Explorer avancé

L'explorateur de fichiers supporte les filtres (Ctrl+F dans l'explorer) et les exclusions via files.exclude dans settings.json. Pour les projets Java, il est utile d'exclure target/ et .class ; pour .NET, bin/ et obj/.

10

Snippets

Les snippets sont des modèles de code déclenchés par un mot-clé. Ils font gagner énormément de temps sur les structures répétitives.

Snippets natifs

VSCode intègre des snippets pour la plupart des langages. En HTML, taper ! puis Tab génère une page HTML5 complète. En JavaScript, clg + Tab génère console.log().

Créer ses propres snippets

Ctrl+Shift+P → Snippets: Configure User Snippets → choisir le langage. Format JSON avec des $1, $2 comme positions du curseur et $TM_FILENAME comme variables.

JSON · snippet Java"Méthode standard": {
  "prefix": "meth",
  "body": [
    "public ${1:void} ${2:nomMethode}(${3:params}) {",
    "    $0",
    "}"
  ],
  "description": "Déclare une méthode Java"
}
💡 Emmet : snippets HTML/CSS intégrés
VSCode intègre Emmet nativement. Taper ul>li*5>a puis Tab génère une liste avec 5 éléments et des liens. En CSS : m10 → margin: 10px. Chercher "Emmet cheat sheet" pour la syntaxe complète.

M2

La palette de commandes

La palette de commandes (Ctrl+Shift+P) est l'outil le plus puissant de VSCode. Elle donne accès à toutes les commandes disponibles — extensions incluses — avec une recherche fuzzy.

🚀 Le préfixe change le mode
Sans préfixe (Ctrl+Shift+P) → toutes les commandes
> → commandes (équivalent à Ctrl+Shift+P)
@ → symboles dans le fichier courant (méthodes, classes...)
# → symboles dans tout le workspace
: → aller à la ligne (équivalent à Ctrl+G)
? → aide sur les modes disponibles

Tout ça est accessible depuis Ctrl+P en tapant le bon préfixe !

Commandes utiles à connaître

M3

Édition multi-curseurs

Les multi-curseurs permettent d'éditer plusieurs endroits simultanément. C'est l'une des fonctionnalités qui différencie les utilisateurs rapides des lents.

🖱 Façons d'obtenir plusieurs curseurs
Alt + clic → ajouter un curseur là où tu cliques
Ctrl+D → sélectionner la prochaine occurrence du mot sélectionné (cumulatif)
Ctrl+Shift+L → sélectionner toutes les occurrences
Alt+Shift + glisser → sélection en bloc (curseurs sur toutes les lignes)
Ctrl+Alt+↑/↓ → ajouter un curseur sur la ligne au-dessus/dessous

Cas d'usage typiques

🔄

Renommer partout

Sélectionner une variable, Ctrl+Shift+L, taper le nouveau nom. Toutes les occurrences sont remplacées.

📋

Modifier plusieurs lignes

Ajouter un préfixe ou suffixe identique sur 10 lignes en une seule opération avec multi-curseurs.

🎯

Alignement de code

Sélectionner en bloc (Alt+Shift+glisser) pour aligner des valeurs sur plusieurs lignes.

M4

Trucs, astuces & bonnes pratiques

Trucs peu connus mais très utiles

💡 Sélectionner jusqu'à une correspondance
Ctrl+M (ou cliquer sur une parenthèse/accolade) puis Ctrl+Shift+M — sélectionne tout le bloc entre parenthèses/accolades. Très utile pour copier/supprimer un bloc entier.
💡 Scroll sans bouger le curseur
Ctrl+↑/↓ scrolle l'éditeur sans déplacer le curseur. Pratique pour consulter du code en haut/bas sans perdre sa position.
💡 Comparer deux fichiers
Dans l'Explorer : clic droit sur un fichier → Select for Compare, puis clic droit sur l'autre → Compare with Selected. Diff visuel immédiat.
💡 Timeline
Dans l'Explorer, en bas, il y a un panneau Timeline : il montre l'historique de sauvegarde locale (même sans Git) et les commits Git du fichier ouvert. Récupérer une version perdue d'un fichier sans Git est possible grâce à ça.

Bonnes pratiques à adopter dès le début

Pratique Pourquoi
Toujours ouvrir un dossier, jamais juste un fichier IntelliSense, Git et les extensions ont besoin du contexte du projet entier
Activer Format on Save dès le début Garde le code lisible automatiquement, évite les disputes de style en équipe
Commiter le dossier .vscode/ dans Git Partager la configuration avec l'équipe, tout le monde a le même environnement
Utiliser F12 au lieu de scroller pour trouver du code Aller à la définition est bien plus rapide que de chercher manuellement
Apprendre Ctrl+P et Ctrl+Shift+P Ces deux raccourcis ouvrent 90% des fonctionnalités sans mémoriser tout le reste
Utiliser le terminal intégré plutôt qu'un terminal séparé Contexte partagé, historique conservé, intégration avec les tasks VSCode

Personnaliser les thèmes

Les thèmes changent plus que la couleur — ils affectent la lisibilité du code pendant des heures. Ctrl+K puis Ctrl+T pour prévisualiser les thèmes installés. Thèmes populaires chez les développeurs : One Dark Pro, GitHub Dark, Dracula, Tokyo Night. Installer via le Marketplace d'extensions.

Pour les icônes de fichiers (qui rendent l'Explorer plus lisible) : Material Icon Theme est la référence.

📱 VSCode sur le Web
vscode.dev est une version entièrement dans le navigateur — sans installation. Pratique pour consulter un repo GitHub rapidement. Sur GitHub, sur n'importe quel repo : appuyer sur la touche . (point) pour ouvrir directement VSCode dans le navigateur.
