Les conventions de nommage

Avant de parler de dossiers, il faut maîtriser les conventions de nommage — le langage universel qui détermine comment s'appellent vos fichiers, classes, méthodes et variables. Ces conventions ne sont pas arbitraires : elles font partie du contrat social de chaque écosystème.

🐍
snake_case
Tout en minuscules, mots séparés par des underscores. Fichiers Python, variables et fonctions en Python.
🐫
camelCase
Première lettre minuscule, chaque mot suivant commence par une majuscule. Variables et méthodes en JavaScript, variables locales en C#.
🏛️
PascalCase
Chaque mot commence par une majuscule. Classes dans tous les langages. Méthodes en C#. Fichiers composants React.
➖
kebab-case
Tout en minuscules, mots séparés par des tirets. URLs, paquets npm, fichiers CSS, dossiers en Node.js.
📢
SCREAMING_SNAKE
Tout en majuscules avec underscores. Constantes dans la majorité des langages (MAX_SIZE, API_KEY).
📦
Dot notation
Mots séparés par des points. Namespaces C# (MonApp.Services) et fichiers de config (app.config.ts).

Python Python

Python impose ses propres règles via la PEP 8. Un dossier devient un paquet dès qu'il contient un __init__.py.

Conventions absolues

  • Fichiers : toujours snake_case.py (jamais UserService.py)
  • Classes : toujours PascalCase
  • Méthodes et fonctions : snake_case()
  • Variables privées : _single_underscore (convention) ou __double (name mangling)
  • Constantes de module : UPPER_SNAKE_CASE
  • Un __init__.py par paquet (même vide)

05

C# — Desktop C#

C# est le langage phare du desktop Windows et cross-platform avec .NET. Selon la technologie desktop utilisée, le pattern architectural change : MVVM pour WPF et MAUI (avec XAML), MVC simplifié ou code-behind direct pour WinForms (sans XAML).

Conventions C# — Ce qui change par rapport aux autres langages

  • Fichiers : toujours PascalCase.cs — nom du fichier = nom de la classe
  • Méthodes : PascalCase() — attention, différent de JS et Python qui utilisent camelCase !
  • Propriétés : PascalCase — public string FirstName { get; set; }
  • Variables locales et paramètres : camelCase
  • Champs privés : _camelCase avec underscore devant
  • Interfaces : préfixe I obligatoire — IUserRepository, IUserService
  • Namespaces : MonApp.Services.Users — reflète le chemin du dossier
🏗️ MVVM vs Code-behind

WPF / MAUI utilisent XAML + le pattern MVVM : la View ne contient que de l'UI, le ViewModel contient la logique. WinForms n'a pas de XAML : l'interface se crée avec un designer visuel, et la logique va souvent directement dans le code-behind du formulaire. C'est plus simple à démarrer, mais plus difficile à maintenir sur un grand projet.

ADO.NET vs Entity Framework — dans vos Repositories

Quand vous accédez à une base de données en C#, vous avez deux approches. Les deux vivent dans vos Repositories/, mais elles fonctionnent très différemment.

ADO.NET — SQL à la main
// Repositories/UserRepository.cs
// ADO.NET = bas niveau, SQL brut
public class UserRepository : IUserRepository
{
  private readonly string _connStr;

  public async Task<User?> FindByIdAsync(int id)
  {
    using var conn = new SqlConnection(_connStr);
    using var cmd  = new SqlCommand(
      "SELECT * FROM Users WHERE Id = @id", conn);
    cmd.Parameters.AddWithValue("@id", id);
    await conn.OpenAsync();
    using var reader = await cmd.ExecuteReaderAsync();
    if (await reader.ReadAsync())
      return new User {
        Id        = reader.GetInt32(0),
        FirstName = reader.GetString(1)
      };
    return null;
  }
}
Entity Framework — ORM moderne
// Data/AppDbContext.cs
public class AppDbContext : DbContext
{
  public DbSet<User> Users { get; set; }
}

// Repositories/UserRepository.cs
// EF = haut niveau, pas de SQL
public class UserRepository : IUserRepository
{
  private readonly AppDbContext _ctx;

  public async Task<User?> FindByIdAsync(int id)
    => await _ctx.Users
         .FindAsync(id);

  public async Task CreateAsync(User user)
  {
    _ctx.Users.Add(user);
    await _ctx.SaveChangesAsync();
  }
}
📁 Structure avec Entity Framework

EF ajoute deux dossiers à votre projet : Data/ pour le AppDbContext.cs (la classe qui représente votre BDD) et Migrations/ pour les fichiers de migration générés automatiquement (l'équivalent des migrations Laravel ou Django). Ces migrations sont créées avec la commande dotnet ef migrations add NomDeLaMigration.

Exemple de code C# bien structuré

C# · Models/User.cs
// Models/User.cs
namespace MonApp.Models;  // namespace reflète le dossier

public class User
{
    // Propriétés : PascalCase
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    public string LastName  { get; set; } = string.Empty;
    public string Email     { get; set; } = string.Empty;

    // Champ privé : _camelCase
    private string _passwordHash = string.Empty;

    // Méthodes : PascalCase (pas camelCase !)
    public string GetFullName() => $"{FirstName} {LastName}";
    public bool IsAdmin() => Role == UserRole.Admin;

    public UserRole Role { get; set; } = UserRole.User;
}

public enum UserRole { User, Admin, Guest }
C# · Services/IUserService.cs + UserService.cs
// Services/IUserService.cs — Interface : préfixe I obligatoire
namespace MonApp.Services;

public interface IUserService
{
    Task<User?> GetByIdAsync(int id);
    Task<IEnumerable<User>> GetAllAsync();
    Task CreateAsync(User user);
}

// Services/UserService.cs
public class UserService : IUserService
{
    private readonly IUserRepository _userRepository; // injection

    public UserService(IUserRepository userRepository)
        => _userRepository = userRepository;

    public async Task<User?> GetByIdAsync(int id)
        => await _userRepository.FindByIdAsync(id);
}

ASP.NET Core C#

ASP.NET Core est le framework web Microsoft en C#. C'est l'équivalent de Laravel (PHP) ou Django (Python), mais pour le web C#. Il supporte deux approches : MVC (Model-View-Controller) pour les sites avec rendu HTML côté serveur, et Web API pour les APIs REST pures (JSON).

🌐 MVC vs Web API — quelle différence ?

ASP.NET MVC : le serveur génère du HTML avec des vues Razor (.cshtml). Comme Laravel avec Blade. ASP.NET Web API : le serveur ne retourne que du JSON. Le frontend (React, Vue, Angular) est séparé. En 2025, le Web API est l'approche la plus courante pour les nouveaux projets.

💡 ViewModel ASP.NET ≠ ViewModel MVVM

Attention à la confusion : en ASP.NET MVC, un ViewModel est juste un objet qui regroupe les données à envoyer à une vue Razor (c'est un DTO spécialisé). Ce n'est pas le même concept que le ViewModel du pattern MVVM (WPF/MAUI). Même nom, rôle différent.

Checklist finale — Suis-je un héros ?

Nommage

  • J'utilise les conventions du langage : snake_case.py, kebab-case.js, PascalCase.cs, PascalCase.php
  • Les noms de classes sont des noms (substantifs), pas des verbes
  • Les noms de méthodes sont des verbes (GetUser() en C#, get_user() en Python)
  • Les interfaces C# commencent par I — IUserRepository, IUserService
  • Les champs privés C# commencent par _ — _userRepository
  • Les noms sont assez précis : UserService > Manager

Structure

  • Un fichier = une classe principale
  • Le nom du fichier correspond exactement au nom de la classe
  • Les dossiers reflètent la responsabilité, pas le type technique seul
  • Le code source est isolé dans src/, les tests dans tests/
  • Les tests suivent la même structure que le source (principe du miroir)
  • Les fichiers de configuration sont à la racine (.env, package.json, appsettings.json)

Architecture

  • La logique métier n'est pas dans les contrôleurs (SRP)
  • Les contrôleurs ne font qu'orchestrer (appeler services, formater réponse)
  • Les repositories cachent les détails d'accès aux données
  • Les interfaces précèdent les implémentations concrètes
  • En C# desktop MVVM : les ViewModels ne contiennent pas de code UI (code-behind minimal)
  • En C# desktop MVVM : la View ne contient que du XAML, zéro logique

Qualité

  • Chaque classe a une seule raison de changer (SRP)
  • Des tests existent et couvrent au minimum les cas nominaux
  • Un README.md explique comment lancer le projet
  • Un .env.example sans valeurs réelles est commité
  • Le .gitignore ignore .env, node_modules/, __pycache__/, bin/, obj/
🏆 Vous êtes un héros si…

Un développeur inconnu peut cloner votre repo, lire l'arborescence 30 secondes, et comprendre immédiatement où chercher n'importe quelle fonctionnalité. Le code n'est pas écrit pour les machines — il est écrit pour les humains.
