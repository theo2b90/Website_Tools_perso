Pourquoi UML ?

Le contexte avant le code

🗺️ UML en une page
▼

Quand on développe seul un petit script, pas besoin de plan. Mais dès qu'on construit une application réelle — avec plusieurs modules, plusieurs développeurs, ou qui doit évoluer dans le temps — il faut parler de l'architecture avant d'écrire la première ligne de code.

UML (Unified Modeling Language) est un langage visuel standardisé pour décrire des systèmes logiciels. Créé dans les années 1990 par Grady Booch, Ivar Jacobson et James Rumbaugh, il est normalisé par l'OMG depuis 1997.

UML n'est pas un langage de programmation. C'est un langage de communication entre développeurs, architectes, chefs de projet et clients.

💡
Analogie clé : UML est aux logiciels ce que les plans d'architecte sont aux bâtiments. Un plan de façade, un plan de sol, un plan électrique — ce sont des vues différentes du même bâtiment. UML fonctionne pareil.

Deux grandes familles

FamilleQuestion poséeDiagrammes
StructureComment est construit le système ?Classes, Objets, Composants, Déploiement, Packages…
ComportementComment se comporte le système ?Cas d'utilisation, Séquence, Activité, État, Communication…

Quand utiliser UML ?

  • Avant de coder : conception (diagrammes de classes, cas d'utilisation)
  • Pendant le dev : communication en équipe, documentation technique
  • Après : rétro-ingénierie d'un système existant pour le comprendre
✅
On n'a pas besoin de maîtriser les 14 diagrammes. En pratique, 5 ou 6 suffisent pour 90 % des projets. Ce cours vous apprendra lesquels et dans quel ordre.

Fondations

Dev junior — Comprendre les bases, lire un diagramme, produire ses premiers schémas

Débutant
🎯 1.1 — Le diagramme de cas d'utilisation
▼

Le diagramme de cas d'utilisation (Use Case Diagram) est souvent le premier diagramme qu'on trace pour un projet. Il répond à : "Qui fait quoi avec le système ?"

Éléments fondamentaux

SymboleNomDescription
🧍 BonhommeActeurPersonne ou système externe qui interagit avec l'application
⭕ EllipseCas d'utilisationUne fonctionnalité du point de vue de l'utilisateur
□ RectangleFrontière systèmeDélimite ce qui est dans / hors du système
→ AssociationLien acteur–casL'acteur participe au cas d'utilisation
«include»InclusionLe cas A inclut toujours le cas B
«extend»ExtensionLe cas B peut étendre optionnellement le cas A

Exemple : Système de bibliothèque

┌─────────────────────────────────────────────────┐ │ Système Bibliothèque │ │ │ │ 🧍 Adhérent ──── (Rechercher un livre) │ │ │ │ │ ├────────── (Emprunter un livre) │ │ │ │ │ │ │ «include» │ │ │ ↓ │ │ │ (Vérifier dispo) │ │ │ │ │ └────────── (Rendre un livre) │ │ │ │ 🧍 Bibliothécaire ─ (Gérer le catalogue) │ │ ── (Gérer les adhérents) │ └─────────────────────────────────────────────────┘

Conseils de lecture

  • Chaque cas d'utilisation commence par un verbe à l'infinitif : "Rechercher", "Emprunter"…
  • Un acteur peut être un système externe (ex : service de paiement)
  • «include» = obligatoire. «extend» = optionnel/conditionnel
  • Ne pas confondre avec un organigramme : on décrit des besoins, pas des étapes
⚠️
L'erreur classique du débutant : mettre des actions techniques ("accéder à la BDD", "valider le token") dans un diagramme de cas d'utilisation. Ces détails n'appartiennent pas ici — c'est le quoi, pas le comment.

📊 1.2 — Le diagramme de classes (introduction)
▼

Le diagramme de classes est le pilier d'UML. Il modélise la structure statique du système : les entités du domaine, leurs attributs, méthodes et relations.

Anatomie d'une classe

┌─────────────────────┐ │ Voiture │ ← Nom de la classe ├─────────────────────┤ │ - marque : String │ ← Attributs │ - annee : int │ (- = privé) │ + couleur: String │ (+ = public) ├─────────────────────┤ │ + demarrer() : void │ ← Méthodes │ + getMarque(): String│ └─────────────────────┘

Visibilité

SymboleMot-cléAccessible depuis…
+publicPartout
-privateLa classe seulement
#protectedLa classe et ses sous-classes
~packageLe package/module

Exemple concret — Librairie en ligne

┌──────────────┐ ┌──────────────┐ │ Client │ │ Commande │ ├──────────────┤ ├──────────────┤ │- nom: String │ 1 0..* │- date: Date │ │- email:String│─────────│- total: float│ ├──────────────┤ ├──────────────┤ │+seConnecter()│ │+calculer() │ └──────────────┘ └──────┬───────┘ │ contient 1..* ┌──────▼───────┐ │ Livre │ ├──────────────┤ │- titre:String│ │- prix: float │ ├──────────────┤ │+getInfos() │ └──────────────┘
📖
Multiplicités : 1 = exactement un, 0..* = zéro ou plusieurs, 1..* = un ou plusieurs, 0..1 = zéro ou un (optionnel).

🏗️ 2.1 — Diagramme de classes avancé
▼

Maintenant que les bases sont posées, on entre dans les relations avancées entre classes : héritage, composition, agrégation, dépendance, réalisation.

Les 5 types de relations

RelationNotationSignificationExemple
Association──Lien simple entre deux classesClient passe Commande
Agrégation◇──"A contient B" mais B peut exister sans AÉquipe contient Joueur
Composition◆──"A contient B" et B n'existe que dans AMaison contient Pièce
Héritage──▷B est une spécialisation de AChat hérite d'Animal
Réalisation╌╌▷B implémente l'interface ADuck implements Flyable
Dépendance╌╌▶A utilise B ponctuellementCommande utilise Taxe

Composition vs Agrégation — La différence clé

📖
Composition (losange plein ◆) : cycle de vie partagé. Si la Voiture est détruite, le Moteur l'est aussi. Sans voiture, pas de moteur.

Agrégation (losange vide ◇) : cycle de vie indépendant. Une Équipe peut être dissoute et les Joueurs existent toujours.

Héritage et interfaces — Exemple complet

┌──────────────┐ │ «interface»│ │ Payable │ └──────┬───────┘ △ (réalisation) ┌──────────┴──────────┐ │ │ ┌────▼─────┐ ┌─────▼────┐ │ Commande │ │ Abonnement│ └──────────┘ └──────────┘ ┌───────────┐ │ Animal │ ← classe abstraite ├───────────┤ │# nom │ │+ manger() │ └─────┬─────┘ △ (héritage) ┌────────┴────────┐ ┌────▼────┐ ┌────▼────┐ │ Chien │ │ Chat │ │+aboyer()│ │+miauler()│ └─────────┘ └─────────┘

Classes abstraites et interfaces

  • Classe abstraite : nom en italique, ne peut pas être instanciée directement, peut avoir des méthodes implémentées
  • Interface : stéréotype «interface», que des méthodes abstraites (en Java/C#), implémentée via réalisation

PlantUML — Relations avancées

@startuml
abstract class Animal {
  # nom : String
  + {abstract} faireDuBruit()
}

interface Domestique {
  + accepteMailleresse() : bool
}

class Chien extends Animal implements Domestique {
  + faireDuBruit() : "Wouf"
  + accepteMailleresse() : true
}

Animal "1" o-- "0..*" Jouet : possède (agrégation)
Chien  "1" *-- "1"   Coeur : a (composition)
@enduml

📦 2.2 — Diagramme de packages
▼

Le diagramme de packages organise les classes en groupes logiques et montre les dépendances entre ces groupes. Indispensable pour comprendre l'architecture en couches d'une application.

Exemple : Architecture MVC

┌─────────────────────────────────────────┐ │ Application Web │ │ │ │ ┌──────────────┐ │ │ │ «package» │ │ │ │ Vue │ │ │ │ (HTML/CSS) │ │ │ └──────┬───────┘ │ │ │ dépend de │ │ ┌──────▼───────┐ │ │ │ «package» │ │ │ │ Contrôleur │ │ │ │ (Logique) │ │ │ └──────┬───────┘ │ │ │ dépend de │ │ ┌──────▼───────┐ ┌──────────────┐ │ │ │ «package» │ │ «package» │ │ │ │ Modèle │ │ Services │ │ │ │ (Entités) │ │ (Métier) │ │ │ └──────────────┘ └──────────────┘ │ └─────────────────────────────────────────┘

Règle des dépendances

Une dépendance (╌╌▶) entre packages signifie : "ce package utilise des éléments de cet autre package". On cherche à minimiser les dépendances cycliques (A dépend de B qui dépend de A = mauvais signe).

✅
Bonne pratique : Les dépendances doivent aller dans un sens. Dans une architecture en couches : Présentation → Métier → Données. Jamais l'inverse.

🖥️ 2.3 — Diagrammes de composants & déploiement
▼

Ces deux diagrammes s'adressent à l'architecture physique et technique du système. On sort du domaine métier pour entrer dans l'infrastructure.

Diagramme de composants

Modélise les composants logiciels (modules, bibliothèques, services) et leurs interfaces. Un composant expose des interfaces fournies (lollipop ○─) et requiert des interfaces requises (socket ─◑).

Exemple : Application e-commerce

[Frontend React] ─────► [API REST] │ ┌───────────┼──────────┐ ▼ ▼ ▼ [Auth Service] [Cart Service] [Product Service] │ │ │ └───────────┴──────────┘ │ [Base de données PostgreSQL] │ [Cache Redis]

Diagramme de déploiement

Montre la topologie physique : sur quelles machines (nœuds) tournent les composants. Essentiel pour documenter une infrastructure cloud ou microservices.

🗂️ 2.4 — Diagramme d'objets (et les diagrammes rares)
▼

UML définit 14 types de diagrammes. Ce cours couvre ceux utilisés dans 95 % des projets réels. Voici un panorama des diagrammes moins courants, utiles à connaître de nom.

Diagramme d'objets

C'est une photo instantanée du diagramme de classes : au lieu de montrer des classes abstraites, on montre des instances concrètes à un moment précis. Très utile pour valider ou expliquer un exemple.

Classes (abstrait) Objets (concret, à t=0) ┌───────────┐ :Client :Commande │ Client │ → nom = "Alice" ──── date = "2024-01-15" │ Commande │ email = "a@…" total = 42.0 └───────────┘
✅
Utilisez le diagramme d'objets quand votre diagramme de classes est complexe et qu'un exemple concret aiderait à le comprendre.

Avancé
🔄 3.1 — Diagramme de séquence
▼

Le diagramme de séquence montre comment les objets interagissent dans le temps pour réaliser un cas d'utilisation. C'est le diagramme le plus utilisé après le diagramme de classes.

Éléments

ÉlémentDescription
Participant/ActeurColonne verticale représentant un objet ou un acteur
Ligne de vieLigne pointillée verticale sous chaque participant
Message synchroneFlèche pleine → : l'appelant attend la réponse
Message asynchroneFlèche ouverte →> : l'appelant n'attend pas
RéponseFlèche pointillée ‹-- : valeur de retour
ActivationRectangle sur la ligne de vie : durée d'exécution
FragmentBloc conditionnel : alt, opt, loop, par…

Exemple : Connexion à une application

Navigateur Contrôleur Service Auth Base de données │ │ │ │ │─ POST /login─►│ │ │ │ │─validateInput()│ │ │ │───────────────►│ │ │ │ │──findUser(email)─►│ │ │ │◄─────── User ─────│ │ │ │ │ │ ┌───────────────────┐ │ │ │ │ alt [mdp correct] │ │ │ │ │ │─generateToken() │ │ │◄── 200 + token ───│ │ │ ├───────────────────┤ │ │ │ [mdp incorrect] │ │ │ │◄─── 401 Error ────│ │ │ └───────────────────┘ │

Fragments combinés

  • alt : branchement conditionnel (if/else)
  • opt : séquence optionnelle (if sans else)
  • loop : répétition
  • par : exécution parallèle
  • ref : référence vers un autre diagramme

PlantUML — Diagramme de séquence

@startuml
actor Utilisateur
participant "Contrôleur" as C
participant "Auth Service" as A
database   "Base de données" as DB

Utilisateur -> C : POST /login(email, mdp)
C -> A : valider(email, mdp)
A -> DB : findUser(email)
DB --> A : User

alt mdp correct
  A --> C : token JWT
  C --> Utilisateur : 200 OK + token
else mdp incorrect
  A --> C : null
  C --> Utilisateur : 401 Unauthorized
end
@enduml

⚡ 3.2 — Diagramme d'activité
▼

Le diagramme d'activité modélise un flux de travail ou un algorithme. C'est l'équivalent UML de l'organigramme, mais bien plus expressif grâce aux swimlanes (couloirs).

Notation de base

SymboleSignification
● (disque plein)Début (état initial)
⊙ (disque dans cercle)Fin (état final)
□ (rectangle arrondi)Action / activité
◇ (losange)Décision (condition)
─── (barre épaisse)Fork/Join — parallélisme
│ Swimlane │Couloir = responsabilité d'un acteur

Exemple : Processus de commande e-commerce

║ Client ║ Système de paiement ║ Entrepôt ║ ║ ║ ║ ║ ║ ● ║ ║ ║ ║ ↓ ║ ║ ║ ║ [Ajouter panier] ║ ║ ║ ↓ ║ ║ ║ ║ [Valider commande] ║ ║ ║ ↓ ║ ║ ║ ║──────────────►║ [Traiter paiement] ║ ║ ║ ║ ↓ ║ ║ ║ ║ ◇ Paiement OK ? ║ ║ ║ ║ / \ ║ ║ ║ ║ [Oui] [Non] ║ ║ ║ ║ ↓ ↓ ║ ║ ║◄──────────────║ Confirmé Refusé ║ ║ ║ ║ ↓ ║ ║ ║ ║───────────────────────►[Préparer] ║ ║ ║ ║ [Expédier] ║ ║ ║ ║ ↓ ║ ║ ║ ║ ⊙ ║
💡
Fork/Join : Une barre horizontale épaisse avec plusieurs sorties = fork (lancement parallèle). La barre qui rassemble les branches parallèles = join (synchronisation). Exemple : envoyer un email ET notifier par SMS en parallèle.

🔀 3.3 — Diagramme d'états-transitions
▼

Le diagramme d'états (State Machine Diagram) modélise le cycle de vie d'un objet. Pour chaque état, on définit les événements qui déclenchent des transitions.

Éléments

  • État : rectangle arrondi — situation dans laquelle se trouve l'objet
  • Transition : flèche entre états avec syntaxe événement [condition] / action
  • État initial : disque plein ●
  • État final : double cercle ⊙
  • Actions internes : entry/, exit/, do/ dans un état

Exemple : Cycle de vie d'une commande

● ↓ ┌──────────────┐ │ En attente │ │ do/envoyer │ │ email │ └──────┬───────┘ │ paiementReçu ▼ ┌──────────────┐ │ En cours de │◄──────────────────┐ │ préparation │ │ └──────┬───────┘ retourDemandé │ expédié │ ▼ │ ┌──────────────┐ ┌──────┴───────┐ │ Expédiée │ │ En retour │ └──────┬───────┘ └──────────────┘ │ livré ▼ ┌──────────────┐ │ Livrée │ └──────────────┘ │ annulé [délai < 24h] ▼ ┌──────────────┐ │ Annulée │ └──────┬───────┘ ↓ ⊙
✅
Ce diagramme est précieux pour modéliser : des workflows (tickets, commandes), des protocoles réseau, des automates finis, ou des objets complexes comme des lecteurs média (play/pause/stop).

💬 3.4 — Diagramme de communication
▼

Le diagramme de communication (anciennement diagramme de collaboration) est une variante du diagramme de séquence : il montre les mêmes interactions mais en mettant l'accent sur la structure des objets plutôt que l'axe temporel.

Différence avec le diagramme de séquence

SéquenceCommunication
AxeTemporel (vertical)Structural (spatial)
Point fortOrdre des messagesRelations entre objets
UsageFlux d'un scénarioQui parle à qui

Les messages sont numérotés (1:, 1.1:, 1.2:…) pour indiquer l'ordre. Ce diagramme est moins courant mais utile pour montrer le réseau d'interactions dans un système dense.

🧱 4.1 — Modèle 4+1 vues
▼

Le modèle 4+1 vues de Philippe Kruchten (1995) est un cadre architectural fondamental qui organise la description d'un système selon 5 perspectives complémentaires.

Les 5 vues

VueDiagrammes UMLAudienceQuestion posée
LogiqueClasses, Objets, SéquenceDéveloppeursQuelles sont les abstractions ?
ProcessActivité, Séquence, ÉtatsIntégrateursComment les processus communiquent ?
DéveloppementPackages, ComposantsDev / Archi.Comment est organisé le code ?
PhysiqueDéploiementDevOps / SREComment est déployé le système ?
+1 ScénariosCas d'utilisationTousQuels sont les cas principaux ?
💡
La vue "+1" (scénarios) sert à valider les quatre autres vues. Si un cas d'utilisation ne peut pas être réalisé avec les éléments des 4 vues, il manque quelque chose dans l'architecture.

🧩 4.2 — Patterns de conception en UML
▼

Les design patterns (patrons de conception) sont des solutions éprouvées à des problèmes récurrents. UML est le langage standard pour les documenter et les communiquer.

Les 3 familles (GoF)

FamilleExemples de patternsProblème résolu
CréationSingleton, Factory, Builder, PrototypeComment créer les objets
StructureAdapter, Decorator, Facade, CompositeComment composer les objets
ComportementObserver, Strategy, Command, IteratorComment les objets communiquent

Exemple : Pattern Observer en UML

🌐 4.3 — UML et microservices
▼

UML reste pertinent dans les architectures modernes. Pour les microservices, certains diagrammes deviennent critiques pour documenter un système distribué complexe.

C4 Model — Complément moderne à UML

Le C4 Model de Simon Brown est complémentaire à UML pour les architectures modernes. Il propose 4 niveaux de zoom :

  • C1 — Context : le système dans son environnement (acteurs, systèmes externes)
  • C2 — Container : les gros blocs techniques (apps, BDD, services)
  • C3 — Component : l'intérieur d'un container
  • C4 — Code : diagramme de classes UML
💡
C4 et UML sont complémentaires : C4 pour la vue macro (stratégique), UML pour la vue micro (technique). Les diagrammes C1/C2/C3 n'ont pas d'équivalent direct en UML standard.

🔁 4.4 — UML dans un processus Agile
▼

UML peut sembler en contradiction avec l'agilité ("pas de documentation inutile"). En réalité, UML léger est parfaitement compatible avec Scrum et Kanban — à condition de l'utiliser de façon pragmatique.

Principe du juste-assez (JITD)

Just-In-Time Design : ne dessiner un diagramme que quand il apporte une valeur claire, et le garder à jour uniquement s'il reste utile.

Quand dessiner un diagramme en sprint ?

  • Sprint Planning : diagramme de séquence pour les US complexes avec beaucoup d'acteurs
  • Refinement : diagramme de classes pour valider un modèle de données
  • Architecture Decision Records (ADR) : diagramme de composants pour documenter un choix d'architecture
  • Onboarding : diagramme de déploiement pour qu'un nouveau dev comprenne l'infra

UML as Code — La bonne pratique

Stocker les diagrammes dans le dépôt Git sous forme de fichiers PlantUML ou Mermaid. Ils évoluent avec le code, sont revus en PR, et toujours synchronisés.

# Dans votre repo Git :
docs/
  architecture/
    overview.puml
    auth-sequence.puml
    domain-classes.puml
  adr/
    001-choix-postgresql.md

  3. Séquence sur les US complexes de votre projet actuel
  4. Diagramme d'activité pour un workflow métier

🟣 Développeur confirmé — 2 mois

  5. Objectif : Produire un dossier d'architecture complet pour une mise en production
