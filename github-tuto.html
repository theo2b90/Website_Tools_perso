Les Fondations

📖 Théorie + pratique 🎯 Objectif : Git configuré & concepts maîtrisés

Git vs GitHub — quelle différence ?

C'est la première confusion à lever. Git est un logiciel qui tourne sur ta machine, il suit l'historique de tes fichiers. GitHub est un site web qui héberge tes dépôts Git en ligne et ajoute des outils collaboratifs par-dessus.

Git → moteur de versioning (local, sur ton PC) GitHub → plateforme cloud (remote, sur internet) + interface web, Issues, Pull Requests, Actions…

Analogie : Git = ton appareil photo. GitHub = l'album photo en ligne que tu partages avec tes amis.

💡

Il existe des alternatives à GitHub : GitLab (souvent auto-hébergé en entreprise) et Bitbucket (intégré à l'écosystème Atlassian). Tout ce que tu apprends ici s'applique aux deux, la syntaxe Git étant identique.

Les concepts fondamentaux

Avant de toucher au terminal, assimile ces 5 concepts — tout le reste en découle.

📁
Repository (repo)

Un dossier de projet suivi par Git, contenant tout l'historique.

📸
Commit

Une photo de tes fichiers à un instant T, avec un message explicatif.

🌿
Branch

Une ligne de développement parallèle. main est la branche principale.

☁️
Remote

La version du repo hébergée sur GitHub. S'appelle conventionnellement origin.

🔁
Push / Pull

Envoyer ses commits vers le remote / récupérer ceux des autres.

Installation de Git

macOS

bash
# Option 1 — Homebrew (recommandé)
brew install git

# Option 2 — Xcode Command Line Tools
xcode-select --install

Windows

Télécharge Git for Windows sur git-scm.com/download/win. L'installateur inclut Git Bash (terminal Linux-like) — utilise-le pour ce cours.

Linux (Debian/Ubuntu)

bash
sudo apt update && sudo apt install git

Vérifie l'installation :

bash
git --version
# → git version 2.x.x

Configuration initiale

Git doit savoir qui tu es pour signer tes commits. À faire une seule fois :

bash
git config --global user.name "Ton Prénom Nom"
git config --global user.email "ton@email.com"

# Éditeur par défaut (VS Code recommandé)
git config --global core.editor "code --wait"

# Nom de branche par défaut (moderne)
git config --global init.defaultBranch main

# Afficher la config
git config --list
⚠️

L'email doit être le même que celui de ton compte GitHub pour que tes commits soient associés à ton profil.

Créer un compte GitHub

Rends-toi sur github.com et crée un compte. Quelques conseils :

  • Choisis un nom d'utilisateur professionnel — il sera visible sur toutes tes contributions open source.
  • Active l'authentification à deux facteurs (Settings → Password and authentication).
  • Configure une clé SSH pour ne pas retaper ton mot de passe à chaque push (voir ci-dessous).

Configurer SSH (recommandé)

bash
# 1. Générer une clé SSH
ssh-keygen -t ed25519 -C "ton@email.com"
# Appuie sur Entrée 3 fois (emplacement par défaut, pas de passphrase)

# 2. Copier la clé publique
cat ~/.ssh/id_ed25519.pub
# Copie tout le texte qui s'affiche

# 3. Sur GitHub : Settings → SSH and GPG keys → New SSH key
# Colle la clé, donne-lui un nom (ex: "Mon PC")

# 4. Tester la connexion
ssh -T git@github.com
# → Hi username! You've successfully authenticated.
🛠 Exercice 1

Setup complet

  1. Installe Git et vérifie la version.
  2. Configure user.name, user.email et init.defaultBranch.
  3. Crée un compte GitHub si ce n'est pas déjà fait.
  4. Configure une clé SSH et teste la connexion avec ssh -T git@github.com.
  5. Lance git config --list — tu dois voir tes infos.

Module 2 · ~1h

Ton Premier Dépôt

📖 Théorie + pratique 🎯 Objectif : créer un repo et maîtriser le cycle add/commit/push

Créer un dépôt sur GitHub

Sur GitHub, clique sur le bouton vert New (ou le + en haut à droite → "New repository"). Remplis :

  • Repository name : mon-premier-projet (pas d'espaces, tirets OK)
  • Description : optionnel mais utile
  • Visibilité : Public ou Private
  • ☑ Add a README file
  • ☑ Add .gitignore → choisis le template selon ton langage

Clique sur Create repository.

Cloner le dépôt en local

Sur la page du repo, clique sur le bouton vert Code et copie l'URL SSH (commence par git@github.com:).

bash
git clone git@github.com:TON_USERNAME/mon-premier-projet.git
cd mon-premier-projet
ls -la
💡

git clone crée un dossier local, télécharge le code, et configure automatiquement origin (le remote) pour toi.

Le cycle de vie d'un fichier

Comprendre les états d'un fichier est crucial — c'est là où les débutants se perdent le plus :

Untracked → (git add) → Staged → (git commit) → Committed ↓ (git push) ↓ Remote (GitHub)
ÉtatSignificationCommande suivante
UntrackedFichier nouveau, pas encore suivi par Gitgit add
ModifiedFichier existant, modifié depuis le dernier commitgit add
StagedDans la "zone de transit" — prêt pour le commitgit commit
CommittedSauvegardé dans l'historique localgit push

Les commandes essentielles

Voir l'état de son dépôt

bash
git status        # Vue d'ensemble des fichiers modifiés
git diff          # Voir les modifications ligne par ligne
git log --oneline # Historique compact des commits

Ajouter des fichiers au staging

bash
git add nom-du-fichier.txt   # Un fichier spécifique
git add .                    # Tous les fichiers modifiés
git add src/                 # Tout un dossier
⚠️

git add . est pratique mais ajoute tout. Vérifie toujours avec git status avant de committer pour ne pas inclure des fichiers secrets (.env, credentials…)

Créer un commit

bash
git commit -m "feat: ajouter la page d'accueil"

# Raccourci : add + commit en une commande (fichiers déjà trackés seulement)
git commit -am "fix: corriger le bug de connexion"
💡

Convention de messages : adopte le format Conventional Commits. Commence par un type : feat: (nouvelle fonctionnalité), fix: (correction), docs: (documentation), refactor:, chore:. Les messages clairs = un historique lisible.

Envoyer sur GitHub

bash
git push                     # Envoie la branche courante
git push origin main         # Explicite : branche main vers origin

# Récupérer les derniers commits de GitHub
git pull                     # = git fetch + git merge

Annuler des erreurs courantes

bash
# Désindexer un fichier ajouté par erreur (avant commit)
git restore --staged fichier.txt

# Annuler les modifications d'un fichier (retour au dernier commit)
git restore fichier.txt

# Modifier le dernier message de commit (AVANT push)
git commit --amend -m "Nouveau message"
🚫

Ne fais jamais git commit --amend ni git push --force sur des commits déjà poussés sur une branche partagée — tu réécrirais l'historique et casseras le travail des autres.

🛠 Exercice 2

Premier workflow complet

  1. Crée un repo mon-premier-projet sur GitHub (avec README).
  2. Clone-le en local.
  3. Crée un fichier notes.md et ajoute du texte dedans.
  4. Fais git status pour voir l'état.
  5. Stage le fichier, puis commit avec un message au format Conventional Commits.
  6. Modifie notes.md, fais un deuxième commit.
  7. Pousse les deux commits sur GitHub avec git push.
  8. Vérifie sur GitHub que tes commits apparaissent.

Module 3 · ~1h30

Les Branches

📖 Théorie + pratique 🎯 Objectif : créer, merger des branches et résoudre des conflits

Pourquoi brancher ?

Une branche est une copie légère de ton code sur laquelle tu travailles sans toucher à main. Tu développes ta fonctionnalité, tu testes, puis tu fusionnes quand c'est prêt.

main ●──────────────────────────────●──── (stable, en production) \ / feature ●──●──●──●──● (ton travail isolé)

Règle d'or : ne jamais committer directement sur main. Toujours passer par une branche.

Créer et naviguer entre les branches

bash
# Lister les branches locales (* = branche courante)
git branch

# Lister aussi les branches distantes
git branch -a

# Créer une nouvelle branche
git branch feature/ma-fonctionnalite

# Basculer sur une branche
git switch feature/ma-fonctionnalite

# Créer ET basculer en une commande (le plus courant)
git switch -c feature/ma-fonctionnalite

# Voir sur quelle branche on est
git branch --show-current
💡

git switch est la commande moderne (Git 2.23+). Tu verras souvent git checkout dans d'anciens tutoriels — c'est l'équivalent, mais switch est plus lisible.

Nommer ses branches

Une convention répandue :

PréfixeUsageExemple
feature/Nouvelle fonctionnalitéfeature/login-oauth
fix/Correction de bugfix/crash-au-demarrage
docs/Documentationdocs/readme-update
refactor/Refactoringrefactor/api-client
chore/Tâches techniqueschore/update-deps

Fusionner une branche (Merge)

bash
# 1. Revenir sur la branche cible (souvent main)
git switch main

# 2. Fusionner la feature branch
git merge feature/ma-fonctionnalite

# 3. Supprimer la branche devenue inutile
git branch -d feature/ma-fonctionnalite          # local
git push origin --delete feature/ma-fonctionnalite  # remote

Résoudre un conflit de merge

Un conflit arrive quand deux branches ont modifié les mêmes lignes d'un même fichier. Git ne sait pas quelle version garder — c'est à toi de décider.

bash
# Git marque les conflits dans les fichiers concernés :
<<<<<<< HEAD
Texte de ta branche main
=======
Texte de la branche feature
>>>>>>> feature/ma-fonctionnalite

Pour résoudre :

  1. Ouvre le fichier conflictuel dans ton éditeur.
  2. Supprime les marqueurs <<<, ===, >>> et garde/fusionne le contenu voulu.
  3. Stage le fichier résolu et crée un commit.
bash
git add fichier-conflictuel.txt
git commit -m "merge: résoudre le conflit sur fichier-conflictuel"
💡

VS Code et IntelliJ ont des interfaces visuelles pour résoudre les conflits. Cherche "Merge Conflicts" dans la palette de commandes — bien plus confortable que l'édition manuelle.

Visualiser l'historique

bash
git log --oneline --graph --all
# Exemple de sortie :
# * a1b2c3d (HEAD -> main) merge: feature login
# |\
# | * f4e5d6c feat: ajouter formulaire de login
# | * 7a8b9c0 feat: créer la page login
# |/
# * 1234abc docs: update readme
🛠 Exercice 3

Feature branch workflow

  1. Sur ton repo de l'exercice précédent, crée une branche feature/ajout-section.
  2. Ajoute un nouveau fichier section.md sur cette branche, avec 3 commits.
  3. Reviens sur main et modifie notes.md (simule un commit concurrent).
  4. Fusionne feature/ajout-section dans main.
  5. Si conflit : résous-le, puis finalise le merge.
  6. Affiche l'historique avec git log --oneline --graph --all.
  7. Supprime la branche feature locale et remote.

Module 4 · ~1h30

GitHub au Quotidien

📖 Interface web + terminal 🎯 Objectif : maîtriser README, Issues et Pull Requests

Un README qui en impose

Le README.md est la première chose que les gens voient sur ton repo. Il s'écrit en Markdown.

bash
# Titre du projet

Description courte et claire en 1-2 phrases.

## Installation
```bash
npm install
npm start
```

## Utilisation
Exemple concret d'utilisation...

## Contribution
Voir [CONTRIBUTING.md](CONTRIBUTING.md)

## Licence
MIT

Les éléments Markdown les plus utiles :

bash
# H1, ## H2, ### H3          → titres
**gras**, *italique*          → emphase
`code inline`                 → code
```bash ... ```               → bloc de code
- item / 1. item numéroté     → listes
[texte](url)                  → lien
![alt](url-image)             → image
| col1 | col2 |               → tableau
---                           → séparateur

Le fichier .gitignore

Dit à Git d'ignorer certains fichiers : dépendances, secrets, fichiers de build, logs…

bash
# Exemple .gitignore pour un projet Node.js
node_modules/
dist/
.env
.env.local
*.log
.DS_Store
.vscode/
💡

gitignore.io génère des .gitignore adaptés à ton stack (Node, Python, Java…). Si tu as déjà commité un fichier par erreur, utilise git rm --cached fichier pour le désindexer sans le supprimer.

Issues — gérer le travail à faire

Les Issues sont le système de tickets de GitHub. Sers-t'en pour : bug reports, demandes de fonctionnalités, tâches, questions.

Interface web → onglet Issues → New issue. Un bon template :

bash
## 🐛 Description du bug
Quand je clique sur "Connexion", la page plante.

## Reproduction
1. Aller sur /login
2. Saisir n'importe quel email
3. Cliquer sur "Se connecter"
4. Voir l'erreur 500

## Comportement attendu
L'utilisateur est redirigé vers /dashboard

## Environnement
- OS: macOS 14
- Node: 20.x

Tu peux référencer une issue dans un commit avec fix #12 — GitHub fermera automatiquement l'issue à la fusion.

Pull Requests — la pièce maîtresse

Une Pull Request (PR) est une demande de fusion d'une branche dans une autre, avec un espace de discussion et de review. C'est le cœur du workflow collaboratif.

Sur GitHub, après avoir poussé une branche :

  1. GitHub affiche une bannière "Compare & pull request" → clique dessus.
  2. Remplis le titre (format Conventional Commits), la description (ce qui change, pourquoi, comment tester).
  3. Assigne des Reviewers si tu travailles en équipe.
  4. Ajoute des Labels (bug, enhancement, help wanted…).
  5. Clique Create pull request.
✅

Un bon titre de PR : feat(auth): ajouter la connexion OAuth Google (#23) — type, scope, description, numéro d'issue liée.

Merger une PR

GitHub propose 3 stratégies de merge :

StratégieRésultatQuand l'utiliser
Merge commitConserve tous les commits + un commit de mergeProjet où l'historique complet compte
Squash & mergeCondense tous les commits en un seulPRs avec beaucoup de "wip: fixes" parasites
Rebase & mergeRejoue les commits sur main sans commit de mergeHistorique linéaire propre

Pour un projet solo ou débutant : Squash & merge est souvent le plus propre.

🛠 Exercice 4

PR complète de bout en bout

  1. Crée une Issue sur ton repo : "Ajouter une section Résumé au README".
  2. Crée la branche feature/readme-resume.
  3. Améliore ton README.md avec du vrai Markdown (titres, listes, code…).
  4. Pousse la branche et ouvre une Pull Request qui référence ton Issue (Closes #1).
  5. Observe la PR sur GitHub : onglets Commits, Files changed.
  6. Merge avec "Squash & merge".
  7. Vérifie que l'Issue s'est fermée automatiquement.

Module 5 · ~1h30

Collaboration & Open Source

📖 Interface web + terminal 🎯 Objectif : contribuer à un projet externe, faire une code review

Fork — contribuer sans permissions

Tu ne peux pas pousser directement sur le repo d'un inconnu. Le fork crée une copie du repo dans ton compte GitHub, sur laquelle tu as tous les droits.

Projet original (upstream) → Fork dans ton compte → Clone sur ton PC ↑ ↓ ↓ └──── Pull Request ────── ta branche ─── git push ────────┘
bash
# 1. Forker via l'interface GitHub (bouton "Fork")

# 2. Cloner TON fork
git clone git@github.com:TON_USERNAME/projet-forkee.git
cd projet-forkee

# 3. Ajouter le repo original comme remote "upstream"
git remote add upstream git@github.com:AUTEUR_ORIGINAL/projet-forkee.git

# Vérifier les remotes
git remote -v
# origin    git@github.com:TON_USERNAME/... (fetch/push)
# upstream  git@github.com:AUTEUR_ORIGINAL/... (fetch/push)

Garder son fork à jour

bash
# Récupérer les nouveaux commits de l'upstream
git fetch upstream

# Les intégrer dans ta branche main locale
git switch main
git merge upstream/main

# Mettre à jour ton fork sur GitHub
git push origin main

Workflow de contribution

bash
# Toujours partir d'un main à jour
git switch main
git pull upstream main

# Créer une branche dédiée
git switch -c fix/correction-typo

# Travailler, committer...
git add .
git commit -m "fix: corriger la typo dans CONTRIBUTING.md"

# Pousser sur TON fork
git push origin fix/correction-typo

# → Ouvrir la PR sur GitHub depuis TON fork vers le repo original
💡

Avant de contribuer, lis toujours le fichier CONTRIBUTING.md du projet (s'il existe). Il indique les conventions de code, comment lancer les tests, etc. Ne pas le lire est la première raison pour laquelle les PRs sont refusées.

Faire une code review

Sur GitHub, onglet Files changed d'une PR :

  • Clique sur le + à gauche d'une ligne pour commenter.
  • Utilise Suggest changes pour proposer une modification directement dans le code.
  • En haut, clique sur Review changes → choisir Comment, Approve ou Request changes.

Bonnes pratiques de review :

  • Sois constructif, pas condescendant. Préfère "Ici, on pourrait utiliser X pour gagner en lisibilité" à "Ce code est mauvais".
  • Distingue les blockers (doit changer) des suggestions (serait mieux). Préfixe tes commentaires : nit: (nitpick), blocker:, idea:.
  • Complimente le bon travail !

Collaborer sur un repo en équipe

Pour un projet où tu travailles avec des collègues (pas en open source) :

  1. Le propriétaire va dans Settings → Collaborators → Add people.
  2. Les collaborateurs clonent directement le repo (pas besoin de fork).
  3. Chacun crée sa branche, pousse, et ouvre des PRs.
✅

Protection de branche (Settings → Branches → Branch protection rules) : oblige à passer par une PR et à avoir au moins 1 approbation avant de merger sur main. Indispensable en équipe.

🛠 Exercice 5

Simuler une contribution

  1. Fork le repo public github.com/nicklockwood/SwiftFormat (ou n'importe quel projet avec un bon README).
  2. Clone ton fork, configure le remote upstream.
  3. Crée une branche docs/fix-example.
  4. Fais une modification mineure (ex: corriger une typo dans README si tu en trouves une, sinon ajoute un commentaire dans un exemple).
  5. Pousse sur ton fork — n'ouvre pas la PR sur l'original pour cet exercice, reste dans ton fork.
  6. Ouvre la PR de ta branche vers le main de ton fork.
  7. Laisse un commentaire de review sur ta propre PR (exercice de prise en main).

Module 6 · ~2h

Workflow Professionnel

📖 Terminal + interface GitHub 🎯 Objectif : rebase, tags, releases, et première GitHub Action

Rebase — l'alternative au merge

Le rebase rejoue tes commits sur une autre branche, produisant un historique linéaire et plus lisible qu'un merge avec commit de fusion.

Avant rebase : Après rebase : main ●──A──B main ●──A──B──C'──D' \ ↑ feature C──D (feature rejouée sur B)
bash
git switch feature/ma-branche

# Rebase sur main (rejouer mes commits sur le dernier main)
git rebase main

# Si conflit pendant le rebase :
# 1. Résoudre le fichier conflictuel
# 2. git add fichier-corrige.txt
# 3. git rebase --continue   (ou --abort pour annuler)

# Pousser après un rebase (force nécessaire car historique réécrit)
git push --force-with-lease   # plus sûr que --force
⚠️

Ne jamais rebaser une branche partagée (genre main ou une branche sur laquelle d'autres travaillent). Le rebase réécrit l'historique — sur une branche solo, c'est parfait.

Rebase interactif — nettoyer son historique

L'outil le plus puissant pour réécrire proprement ses commits avant une PR :

bash
# Modifier les 4 derniers commits
git rebase -i HEAD~4

# Dans l'éditeur qui s'ouvre, pour chaque ligne :
# pick   → garder tel quel
# reword → changer le message
# squash → fusionner avec le commit précédent
# drop   → supprimer ce commit

Tags et Releases

Les tags marquent des points importants dans l'historique — typiquement des versions. La convention semver : v1.2.3 (major.minor.patch).

bash
# Créer un tag annoté
git tag -a v1.0.0 -m "Version 1.0.0 — première release stable"

# Pousser le tag sur GitHub
git push origin v1.0.0

# Pousser tous les tags
git push --tags

# Lister les tags
git tag -l

Sur GitHub, va dans Releases → Draft a new release → choisis ton tag → rédige les notes de version (changelog). Les releases permettent d'attacher des binaires téléchargeables.

GitHub Actions — CI/CD en 10 minutes

GitHub Actions automatise des tâches : tester ton code à chaque push, déployer sur merge, envoyer des notifications…

Un workflow est un fichier YAML dans .github/workflows/.

yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Récupérer le code
        uses: actions/checkout@v4

      - name: Installer Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Installer les dépendances
        run: npm ci

      - name: Lancer les tests
        run: npm test

      - name: Linter
        run: npm run lint
💡

Le Marketplace GitHub Actions (github.com/marketplace) référence des milliers d'actions prêtes à l'emploi : déploiement sur AWS/Vercel, envoi de notifications Slack, release automatique…

Anatomie d'un workflow

TermeDescription
on:Déclencheur (push, PR, schedule, manuel…)
jobs:Unités de travail qui s'exécutent en parallèle par défaut
runs-on:Machine virtuelle (ubuntu-latest, windows-latest, macos-latest)
steps:Étapes séquentielles d'un job
uses:Action externe réutilisable
run:Commande shell à exécuter
🛠 Exercice 6

Pipeline CI + première release

  1. Dans ton repo, crée le dossier .github/workflows/.
  2. Crée un fichier ci.yml avec un workflow basique (adapte-le à ton langage : si pas de tests, fais juste un echo "Build OK").
  3. Pousse sur main — va dans l'onglet Actions de ton repo GitHub et regarde ton workflow s'exécuter.
  4. Corrige le workflow si erreur, jusqu'à voir une coche verte ✅.
  5. Crée et pousse un tag v0.1.0.
  6. Sur GitHub, crée une Release à partir de ce tag avec des notes de version.

Module 7 · ~2h

Techniques Avancées & Maîtrise

📖 Terminal avancé 🎯 Objectif : outils de pro, alias, GitHub Pages, projet final

git stash — mettre son travail de côté

Tu travailles sur quelque chose et on te demande d'urgence de fixer un bug sur une autre branche. git stash met tes modifications en suspens sans créer de commit.

bash
git stash                    # Empile les modifs non commitées
git stash push -m "WIP: feature login"  # Avec un nom

git stash list               # Voir la pile de stash
# stash@{0}: On main: WIP: feature login
# stash@{1}: On feature/ui: boutons redesign

git stash pop                # Applique et supprime le dernier stash
git stash apply stash@{1}    # Applique un stash spécifique sans le supprimer
git stash drop stash@{0}     # Supprimer un stash
git stash clear              # Vider toute la pile

git cherry-pick — transplanter un commit

Tu veux appliquer un commit spécifique d'une branche sur une autre, sans tout merger.

bash
# Trouver le hash du commit à transplanter
git log --oneline feature/experimental
# a1b2c3d fix: corriger la validation email

git switch main
git cherry-pick a1b2c3d    # Applique ce commit sur main

git bisect — trouver quel commit a introduit un bug

La recherche dichotomique dans l'historique : indispensable quand un bug est apparu "quelque part" dans 200 commits.

bash
git bisect start
git bisect bad              # Le commit actuel est buggé
git bisect good v2.0.0      # Ce tag fonctionnait bien

# Git checkout automatiquement un commit intermédiaire
# Tu testes → tu dis à Git si c'est bon ou mauvais
git bisect good             # ou
git bisect bad

# Git continue jusqu'à isoler le commit coupable
# À la fin :
git bisect reset            # Revenir à HEAD

git reflog — la machine à remonter le temps

Le reflog enregistre tout ce qui s'est passé localement — même un git reset --hard n'efface pas ces traces. C'est ton filet de sécurité ultime.

bash
git reflog
# HEAD@{0}: reset: moving to HEAD~3
# HEAD@{1}: commit: feat: ajout authentification
# HEAD@{2}: commit: fix: corriger typo

# Récupérer des commits "perdus" après un reset
git checkout HEAD@{1}       # Retourner à ce point
# ou
git reset --hard HEAD@{1}   # Restaurer complètement
✅

Si tu fais un git reset --hard par erreur, ne panique pas. Le reflog garde tout pendant 90 jours. Tes commits ne sont jamais vraiment perdus.

Alias Git — accélérer son workflow

Configure des raccourcis dans ~/.gitconfig :

bash
git config --global alias.st "status"
git config --global alias.co "switch"
git config --global alias.br "branch"
git config --global alias.lg "log --oneline --graph --all --decorate"
git config --global alias.undo "reset HEAD~1 --mixed"
git config --global alias.save "!git add -A && git commit -m 'SAVEPOINT'"

# Utilisation
git st        # git status
git lg        # historique graphique
git undo      # défaire le dernier commit (garde les fichiers)

GitHub Pages — héberger gratuitement

GitHub Pages déploie un site statique directement depuis ton repo. Idéal pour : portfolio, documentation, landing page, blog.

  1. Repo → Settings → Pages
  2. Source : Deploy from a branch
  3. Branch : main, dossier : /(root) ou /docs
  4. Ton site sera disponible sur username.github.io/nom-du-repo
💡

Pour un site plus élaboré, utilise GitHub Actions pour déployer automatiquement un framework (Next.js, Astro, Hugo…) sur GitHub Pages à chaque push.

🛠 Exercice Final

Projet complet de bout en bout

  1. Crée un repo portfolio-site avec README, .gitignore, licence MIT.
  2. Configure une Action CI qui vérifie que le HTML est valide (utilise validator/html5validator-action).
  3. Développe sur des branches : crée 3 branches (feature/header, feature/about, feature/contact), fais des commits sur chacune.
  4. Crée 3 PRs, merge-les une par une avec Squash & merge.
  5. Utilise git stash au moins une fois pendant le développement.
  6. Crée un tag v1.0.0 et une Release GitHub.
  7. Active GitHub Pages — ton site doit être accessible en ligne.
  8. Bonus : inviter un ami à fork ton repo et à soumettre une PR.
