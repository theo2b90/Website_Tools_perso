01 — Comprendre Git
Git ≠ GitHub. Git est un logiciel de gestion de versions installé sur ta machine. GitHub est une plateforme en ligne qui héberge des dépôts Git et ajoute des outils collaboratifs (pull requests, issues…). On peut utiliser Git sans GitHub — mais pas l'inverse.
Les 4 zones de Git

Première installation
# Vérifier si Git est installé
git --version

# macOS (via Homebrew)
brew install git

# Ubuntu / Debian
sudo apt install git

# Windows → télécharger Git for Windows
# https://git-scm.com/download/win
Configuration initiale (à faire une fois)

Ces infos apparaissent dans chaque commit. Utilise la même adresse que ton compte GitHub.

git config --global user.name  "Prénom Nom"
git config --global user.email "toi@exemple.com"

# Éditeur par défaut (messages de commit)
git config --global core.editor "code --wait" # VS Code
git config --global core.editor "nano"       # nano

# Branche principale nommée "main" (recommandé)
git config --global init.defaultBranch main

# Voir toute la config
git config --list
03 — Workflow quotidien
Le cycle de base est toujours le même : modifier → vérifier → stager → commiter → pousser. Commiter souvent avec des messages clairs vaut mieux que de tout mettre dans un seul gros commit.
Démarrer un projet
# Option A : nouveau projet local
git init                   # crée le dossier .git/
git init mon-projet       # + crée le dossier

# Option B : cloner un repo existant (GitHub)
git clone https://github.com/user/repo.git
git clone url nom-local  # clone dans un dossier renommé
Observer l'état du repo
# Vue d'ensemble (fichiers modifiés, staged, untracked)
git status
git status -s            # version courte

# Voir les différences
git diff                  # working dir vs index
git diff --staged         # index vs dernier commit
git diff main..feature   # entre deux branches

# Historique des commits
git log
git log --oneline --graph --all  # vue condensée + arbre
git log -p fichier.txt           # historique d'un fichier
Stager & commiter
# Stager des fichiers
git add fichier.txt       # un fichier
git add src/             # un dossier entier
git add .                # tout le working dir
git add -p               # choisir les hunks interactivement

# Désélectionner du staging
git restore --staged fichier.txt

# Commiter
git commit -m "feat: ajoute la page d'accueil"
git commit               # ouvre l'éditeur configuré
git commit --amend      # modifier le dernier commit (avant push)

# Raccourci : stage + commit en une étape (fichiers suivis)
git commit -am "fix: corrige le bug de login"
Synchroniser avec GitHub
# Lier un repo local à GitHub (première fois)
git remote add origin https://github.com/user/repo.git
git remote -v           # vérifier les remotes

# Pousser vers GitHub
git push                 # push branche courante
git push -u origin main # première fois + mémorise
git push origin feature # pousser une branche spécifique

# Récupérer depuis GitHub
git pull                 # fetch + merge
git pull --rebase       # fetch + rebase (historique propre)
git fetch                # télécharge sans merger
git fetch --all         # toutes les branches distantes
04 — Branches
Une branche est juste un pointeur vers un commit — créer une branche est instantané et quasi-gratuit. La règle d'or : ne jamais travailler directement sur main. Toujours créer une branche pour chaque feature ou bugfix.
Gérer les branches
# Lister les branches
git branch               # locales
git branch -a           # locales + distantes

# Créer
git branch feature/login
git switch -c feature/login  # crée + bascule (moderne)
git checkout -b feature/login # ancienne syntaxe

# Basculer entre branches
git switch main
git switch -             # revenir à la branche précédente

# Supprimer
git branch -d feature/login   # si mergée
git branch -D feature/login   # forcer
git push origin --delete feature/login # distante

# Renommer
git branch -m ancien nouveau
Fusionner des branches

Merge préserve l'historique exact. Rebase réécrit les commits sur la cible — historique plus linéaire mais ne jamais rebaser une branche partagée. 

# Merge : se placer sur la cible, fusionner la source
git switch main
git merge feature/login
git merge --no-ff feature/login  # force un merge commit
git merge --squash feature/login # compresse en 1 commit

# Rebase : déplacer la branche sur main
git switch feature/login
git rebase main

# En cas de conflit (merge ou rebase)
# → éditer les fichiers conflictuels, puis :
git add fichier-conflictuel.txt
git merge --continue
# ou abandonner :
git merge --abort
05 — Annuler & Corriger
Règle critique : git reset et git rebase réécrivent l'historique. Ne jamais les utiliser sur des commits déjà poussés sur une branche partagée — ça crée des conflits pour tout le monde. Sur ta branche locale non partagée : tout est permis.
Annuler des modifications
# Restaurer un fichier (working dir → dernier commit)
git restore fichier.txt

# Désélectionner du staging (keep changes in working dir)
git restore --staged fichier.txt

# Supprimer fichiers non suivis
git clean -fd           # fichiers + dossiers
git clean -n            # dry-run (voir ce qui serait supprimé)

# Annuler le dernier commit
# → garde les changements dans staging :
git reset --soft HEAD~1
# → garde les changements dans working dir :
git reset --mixed HEAD~1  # défaut
# → supprime tout (dangereux) :
git reset --hard HEAD~1
Annuler sans réécrire l'historique

git revert crée un nouveau commit qui inverse les changements. Sûr sur les branches partagées. 

# Annuler un commit spécifique (ajoute un commit inverse)
git revert abc1234
git revert HEAD          # annule le dernier commit
git revert HEAD~3..HEAD  # annule les 3 derniers

# Récupérer un fichier depuis un commit précédent
git checkout abc1234 -- fichier.txt

# Retrouver un commit perdu (reflog = filet de sécurité)
git reflog               # tout l'historique de HEAD
git reset --hard HEAD@{2}  # revenir à cet état
Le reflog enregistre tous les mouvements de HEAD, même les reset --hard. C'est le filet de sécurité ultime — les données sont conservées ~90 jours.
Stash — mettre de côté

Utile quand on doit changer de branche avec des modifications en cours non commitées. 

git stash                 # range working dir + index
git stash push -m "WIP login form"  # avec message
git stash -u             # inclut les fichiers non suivis

git stash list            # voir tous les stashs
git stash pop             # réapplique le dernier + supprime
git stash apply stash@{1} # réapplique sans supprimer
git stash drop stash@{0}
git stash clear           # vide tout
Cherry-pick — copier un commit

Applique un commit spécifique sur la branche courante, sans merger toute la branche source. 

# Copier le commit abc1234 sur la branche courante
git cherry-pick abc1234
git cherry-pick abc1234..def5678  # une plage
git cherry-pick -n abc1234       # sans commiter
Tags — marquer une version
git tag v1.0.0              # tag léger
git tag -a v1.0.0 -m "Release 1.0" # annoté
git push origin v1.0.0      # pousser le tag
git push origin --tags      # pousser tous les tags
git tag                     # lister
06 — GitHub — Collaboration
Le workflow standard sur GitHub : fork → branch → commit → pull request → review → merge. Sur un projet en équipe sans fork, on saute l'étape fork : branch directement sur le repo partagé.
Workflow Pull Request (PR)
Contribuer à un projet
# 1. Forker sur GitHub (bouton "Fork")
# 2. Cloner son fork
git clone https://github.com/TOI/repo.git

# 3. Ajouter l'original comme remote "upstream"
git remote add upstream https://github.com/ORIGINAL/repo.git

# 4. Créer une branche
git switch -c feature/ma-contribution

# 5. Faire ses commits, puis pousser
git push -u origin feature/ma-contribution

# 6. Ouvrir la PR sur GitHub (bouton "Compare & pull request")

# 7. Mettre à jour son fork depuis l'original
git fetch upstream
git rebase upstream/main
.gitignore

Fichier à la racine du repo listant les chemins que Git doit ignorer. À créer dès le début — les fichiers déjà suivis ne sont pas rétroactivement ignorés. 

# Dossiers
node_modules/
.next/
dist/
build/

# Fichiers d'environnement (ne jamais commiter !)
.env
.env.local
.env.*.local

# OS et éditeurs
.DS_Store
Thumbs.db
.vscode/
*.log

# Ignorer toutes les extensions .key sauf une
*.key
!public.key

Modèles prêts à l'emploi : gitignore.io ou GitHub propose des templates à la création.

Conventions de commit

Format Conventional Commits — lisible, compatible avec le changelog automatique :

featnouvelle fonctionnalité
fixcorrection de bug
docsdocumentation
styleformatage, pas de logique
refactorrestructuration sans bug fix
testajout ou modif de tests
choremaintenance, dépendances
# Exemples
feat(auth): ajoute le login OAuth
fix: corrige crash sur mobile Safari
docs: met à jour le README
Commandes utiles GitHub CLI

gh est l'outil officiel GitHub en ligne de commande — évite d'aller sur le site pour les opérations courantes. 

# Authentification
gh auth login

# Créer un repo
gh repo create mon-projet --public

# Créer une PR
gh pr create --title "feat: login" --body "..."

# Voir les PRs
gh pr list
gh pr view 42
gh pr checkout 42      # tester une PR localement

# Issues
gh issue create
gh issue list
SSH — authentification

SSH évite de saisir son mot de passe à chaque push. À faire une fois par machine. 

# Générer une clé SSH
ssh-keygen -t ed25519 -C "toi@exemple.com"

# Afficher la clé publique (à coller sur GitHub)
# Settings → SSH keys → New SSH key
cat ~/.ssh/id_ed25519.pub

# Tester la connexion
ssh -T git@github.com

# Passer un remote de HTTPS à SSH
git remote set-url origin
  git@github.com:user/repo.git
07 — Scénarios fréquents
Situations du quotidien
J'ai commité sur main par erreur
# Déplacer le dernier commit sur une nouvelle branche
git branch feature/correction   # crée la branche
git reset --hard HEAD~1         # retire le commit de main
git switch feature/correction  # le commit est là
Je veux mettre à jour ma branche depuis main
git switch feature/login
git fetch origin
git rebase origin/main   # ou merge, selon les habitudes
Résoudre un conflit de merge
# Git marque les conflits ainsi :
<<<<<<< HEAD
mon code actuel
=======
code entrant
>>>>>>> feature/login

# 1. Éditer le fichier pour choisir / fusionner
# 2. Supprimer les marqueurs <<<<, ====, >>>>
git add fichier-conflit.txt
git merge --continue
Autres situations
Push refusé (remote a des commits que je n'ai pas)
git pull --rebase   # intègre les nouveautés, rejoue mes commits
git push             # fonctionne maintenant
Trouver quel commit a introduit un bug (git bisect)
git bisect start
git bisect bad         # HEAD est cassé
git bisect good v1.2  # v1.2 était OK
# Git checkout un commit intermédiaire
# Tester, puis : git bisect good / git bisect bad
git bisect reset       # revenir à HEAD
Voir qui a écrit quelle ligne (git blame)
git blame fichier.txt
git blame -L 10,20 fichier.txt  # lignes 10 à 20
# → chaque ligne montre : hash, auteur, date
08 — GitHub Actions (CI/CD, aperçu)
GitHub Actions automatise des tâches déclenchées par des événements GitHub (push, PR, schedule…). Les workflows sont définis dans .github/workflows/*.yml.
Workflow minimal — tests automatiques
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install & test
        run: |
          npm ci
          npm test
Concepts Actions
workflowfichier YAML = pipeline complet
trigger (on:)push, pull_request, schedule, workflow_dispatch…
jobensemble de steps sur une même machine
stepcommande shell ou action réutilisable
action (uses:)bloc réutilisable depuis le Marketplace
runnermachine virtuelle GitHub (ubuntu/windows/macos)
secretsvariables chiffrées (Settings → Secrets)
artifactsfichiers produits par le workflow (logs, dist…)
